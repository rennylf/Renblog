网络用户评论是理解用户的重要入口。相较于问卷或访谈，评论的获取成本低、覆盖面广，也更贴近日常使用场景。但问题同样明显：评论往往零散、非结构化，价值密度低，真正的分析成本并不低。

近年来，大模型与 AI 聊天机器人的出现，显著提升了研究效率。但在实际工作中我逐渐意识到：聊天机器人更多是在“辅助研究者”，而不是“替代流程”。研究者仍然需要反复拆解任务、组织输入、整合输出。

相比之下，AI Agent 提供了一种新的可能性——它不只是回答问题，而是按照既定目标，持续、自动地完成一系列研究步骤。

本文展示的是基于 Coze 平台，对「车型功能评论分析」这一典型用户研究任务的 AI Agent 化尝试，希望为后续更复杂的 Agent 落地提供一个可复用的参考案例。

设计目标
----

本案例的目标，是分析某一具体车型的用户评论，梳理用户对不同功能模块的评价与关注点。

在传统工作流中，这一任务通常包括：

*   • 收集用户评论
    
*   • 标注评论涉及的功能
    
*   • 按功能聚合评论
    
*   • 综合分析并形成洞察与建议
    

其中，真正消耗精力的并不是“收集”，而是“分类与综合”。研究者需要逐条判断评论提到了哪些功能，再分别对功能相关评论进行总结。这一过程高度依赖人工处理，也很难规模化。

在本案例中，我尝试将这一流程交由 AI Agent 执行。以 Coze 平台为例，研究者只需要：

*   • 上传用户评论数据
    
*   • 指定需要分析的功能模块
    

剩余的检索、聚合、总结与输出，由 Agent 作为“代理研究者”自动完成。

设计思路
----

Agent 的角色并不是“一次性生成结论”，而是按研究逻辑拆解任务，并顺序执行。

![](../images/bqRwJH6uPPaWicaohg640wx_fmtpngfromappmsg.png)

  

1.  1. 数据准备： 研究者上传指定车型的用户反馈数据到 Coze 数据库。
    
2.  2. 功能输入： AI Agent 获取研究者指定的功能名称列表（如“辅助驾驶”、“车机系统”等）。
    
3.  3. 数据查询（循环体）：
    

*   • AI Agent 在数据库中检索当前功能名称相关的所有评论内容。
    

5.  4. 功能分析（循环体）：
    

*   • 将检索到的评论内容输入给大模型（LLM）。
    
*   • LLM 基于研究者的要求（如“总结用户的主要满意点/不满意点”、“提取核心痛点”）进行总结和洞察。
    

7.  5. 结果整合： AI Agent 重复执行 3-4 步，分析完所有功能后，将所有分析结果输入给 LLM。
    
8.  6. 综合输出： LLM 整理并输出最终的结构化分析报告。

一些问题与取舍
-------

### 为什么使用数据库，而不是知识库

在评论分析场景中，研究目标通常是：尽可能覆盖所有相关评论，而不是“找到最相关的几条”。知识库通常基于 RAG 机制返回有限数量的相关结果。例如在 Coze 平台中，单次检索最多支持 20 条内容，这在总结知识点时是足够的，但在评论分析中容易造成样本缺失。

相比之下，数据库的单次返回上限可达 1000 条，更符合“全量分析”的需求。当然，目前也遇到有些评论无法上传的问题，推测可能与字段长度限制有关，这部分仍需进一步验证。

### 使用大模型进行格式化，而不是写代码

在流程中存在不少数据格式转换需求。例如，批处理节点要求输入数组格式，而研究者更自然的输入方式，往往是一行用逗号分隔的功能名称。

理论上可以通过代码节点完成格式转换，但在实际使用中我发现：直接使用大模型节点反而更高效。只需清晰说明转换规则，模型即可完成格式整理，减少了代码维护成本，也降低了研究者的使用门槛。

未来优化方向
------

### 数据库

当前数据库主要是人工收集的测试数据（来自懂车帝平台的领克 900 口碑评论）。后续可以：

*   • 扩展到更多车型与平台
    
*   • 引入自动化的数据采集流程
    
*   • 构建长期可复用的评论分析数据库
    

### 交互界面

目前主要通过对话界面触发分析流程。Coze 还提供 AI 应用形态，允许研究者自定义交互界面，这为后续将 Agent 从“实验工具”发展为“研究应用”提供了可能。
